# Build Plan to Service Catalog Alignment

## Purpose

This document is the crosswalk between the phase-oriented build plan and the per-tool/per-service implementation files in `docs/service_catalog`.

The build plan remains the canonical order of work. The service catalog remains the canonical per-service implementation scope. When a service spans multiple phases, the earliest linked phase is the first implementation point and later linked phases are expansion or hardening points.

## Alignment Rules

- Every service implementation file must keep a `First Build Phase` section that links to the relevant phase document.
- Phase 0 through Phase 13 names must match `docs/build_plan/master_plan.md`.
- A service may appear in more than one phase when it starts as a smaller primitive and expands later.
- Native apps remain ordinary Overrid clients; their service files should point to Phase 12 unless they are governance hardening surfaces that continue into Phase 13.
- Economic services must stay structural and avoid pricing, customer-count, or revenue projection assumptions.
- Per-SDS sub-build plans must preserve the service's first build phase, link back to the numbered SDS, and use master phases only as dependencies or later hardening gates.
